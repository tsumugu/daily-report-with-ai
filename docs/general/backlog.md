# バックログ

**目的**: すぐに開発する予定のアイテムを管理する

**管理方法**: PdM が Discovery の成果物で Adopt と判断したものを Queue する

**更新頻度**: Phase 5（棚卸し）のたびに更新

---

## フォーマット

### アイテムの記録形式

```markdown
## BL-{番号}: {アイテム名}

**優先順位**: P0 / P1 / P2 / P3  
**ステータス**: Queue / In Progress / Done  
**Queue日**: YYYY-MM-DD  
**判断理由**: [Adoptと判断した理由]  
**関連Discovery**: `.discovery/handoff.md` (YYYY-MM-DD)  
**関連問題**: P-{xx} (problem_statements.md)  
**関連要求**: R-{xx} (requirements.md)

### 概要

[アイテムの概要]

### 次のアクション

- [ ] Phase 1: 設計開始
- [ ] PRD作成
- [ ] その他

### 備考

[その他のメモ]
```

### 優先順位

- **P0**: 最優先（すぐに開発を開始）
- **P1**: 高優先度（次の開発候補）
- **P2**: 中優先度（順次開発）
- **P3**: 低優先度（余裕があれば開発）

### ステータス

- **Queue**: バックログにQueueされている（開発未着手）
- **In Progress**: 開発中（Phase 1〜4のいずれか）
- **Done**: 開発完了（Phase 4完了）

### アイテムの追加方法

1. Discovery の成果物で **Adopt** と判断されたアイテムを追加
2. 優先順位（P0〜P3）を決定
3. ステータスを **Queue** に設定
4. 関連情報（Discovery、問題、要求）を記録

### ステータスの更新

- **Queue → In Progress**: Phase 1（設計）開始時に更新
- **In Progress → Done**: Phase 4（完了）完了時に更新

## BL-01: 目標階層管理機能

**優先順位**: P1  
**ステータス**: Done  
**Queue日**: 2025-12-13  
**完了日**: 2025-12-16  
**判断理由**: Discoveryの結果、目標の階層構造を定義・管理する機能は他の機能の前提となる基盤機能として最優先で開発すべきと判断。既存ロードマップの「Phase 3-2：中長期目標との接続機能」と整合性があり、優先度を上げて検討する価値がある。  
**関連Discovery**: `.discovery/handoff.md` (2025-12-13)  
**関連問題**: P-01 (problem_statements.md)  
**関連要求**: R-01 (requirements.md)  
**既存ロードマップ**: Phase 3-2（中長期目標との接続機能）を統合

### 概要

目標を階層的に定義・管理する機能。長期目標から中期目標、短期目標へと階層的にブレイクダウンできる。目標間の関係性（因果関係、依存関係）を明確に可視化し、目標の性質に応じたブレイクダウン方法をサポートする。

**主な機能**:

- 目標を階層的に定義・管理（階層数は柔軟に設定可能、基本は2〜3階層）
- 目標の期間設定は柔軟に対応（半期、四半期、1ヶ月、1週間など、目標の性質に応じて設定可能）
- 目標間の階層構造を一元的に管理
- 目標間の関係性（因果関係、依存関係）を明確に可視化
- 目標の性質に応じたブレイクダウン方法をサポート（例: スキル習得、プロジェクト完了、習慣形成など）
- 既存の週次フォーカス機能との接続

## BL-03: 日報と目標の関連付け機能

**優先順位**: P1  
**ステータス**: Queue  
**Queue日**: 2025-12-13  
**判断理由**: 日々の活動と目標を接続することで、目標達成への道筋が明確になる。ユーザー価値が高い。  
**関連Discovery**: `.discovery/handoff.md` (2025-12-13)  
**関連問題**: P-02 (problem_statements.md)  
**関連要求**: R-02 (requirements.md)

### 概要

日報の内容（やったこと、よかったこと、改善点）と目標を関連付ける機能。関連付けの負担を軽減する仕組み（タグ付け、自動関連付けなど）を提供する。

**主な機能**:

- 日報の内容と目標を関連付けることができる
- 関連付けの負担を軽減する仕組み（例: タグ付け、自動関連付けの検討など）
- 日々の活動がどの目標の達成に貢献しているかを明確にできる
- 目標達成のための具体的なアクションが日報から抽出できる
- 日報と目標の関連性を可視化できる
- 複数の目標に関連付けられた場合、目標ごとの貢献度を定量化できる（任意）

## BL-06: 目標別取り組み集約表示機能

**優先順位**: P1
**ステータス**: Queue
**Queue日**: 2025-12-16
**判断理由**: 半期ごとの振り返りという具体的なユースケースに対応し、自己評価や上司への報告に直結する実用的な機能。目標に関連する取り組みを一箇所に集約することで、振り返りの効率と質が大幅に向上する。実装の複雑度は比較的低く、即座に価値を提供できる。
**関連Discovery**: `.discovery/handoff.md` (2025-12-16)
**関連問題**: P-01 (problem_statements.md)
**関連要求**: R-01 (requirements.md)

### 概要

目標を選択すると、その目標に関連付けられた日報、フォローアップ、週次フォーカスを集約して一覧表示する機能。期間を指定して特定期間の取り組みを確認でき、取り組みの種類でフィルタリングできる。振り返りに必要な情報を一箇所に集約し、すぐにアクセスできるようにする。

**主な機能**:

- 目標を選択すると、関連する取り組み（日報、フォローアップ、週次フォーカス）が一覧表示される
- 期間を指定して、特定期間の取り組みを表示できる（例: 半期、四半期、カスタム期間）
- 取り組みの種類（日報、フォローアップ、週次フォーカス）でフィルタリングできる
- 各取り組みの詳細（日付、内容、ステータスなど）が表示される
- 取り組みをクリックすると、詳細画面に遷移できる
- 取り組みの総数や概要（サマリー）が表示される

### 次のアクション

- [ ] Phase 1: 設計開始（BL-03, BL-04完了後）
- [ ] PRD作成

### 備考

- **前提条件**: BL-03（日報と目標の関連付け機能）とBL-04（フォローアップと目標の接続機能）が実装されていることが必須
- 週次フォーカスと目標の関連付けは、BL-01（目標階層管理機能）で「既存の週次フォーカス機能との接続」が含まれているため、それを前提とする
- 期間指定は柔軟に設定できるようにする（半期、四半期、月次、カスタム期間など）
- 表示順序は時系列（新しい順/古い順）やデータ種別で並び替えられるようにする
- 目標が階層構造を持つ場合、表示オプションを提供する（「選択した目標のみ」「子目標を含む」「親目標を含む」など）

## BL-07: アクティビティタイムライン機能

**優先順位**: P1
**ステータス**: Queue
**Queue日**: 2025-12-16
**判断理由**: 全体俯瞰という広範なユースケースに対応し、成長の実感やモチベーション向上につながる。類似サービス（Notion、Obsidian等）でも採用されている一般的なアプローチであり、ユーザー価値が高い。時系列でのパターンや傾向の発見を支援する。
**関連Discovery**: `.discovery/handoff.md` (2025-12-16)
**関連問題**: P-02 (problem_statements.md)
**関連要求**: R-02 (requirements.md)

### 概要

目標、日報、フォローアップ、週次フォーカスを時系列で統合表示する機能。「いつ、何のために（どの目標のために）、何をしたか」が一目で分かるタイムライン形式で表示する。期間を指定して特定期間のタイムラインを表示でき、データ種別や目標でフィルタリングできる。活動の流れや変化を視覚的に把握できるようにする。

**主な機能**:

- 目標、日報、フォローアップ、週次フォーカスを時系列で統合表示
- 期間を指定して、特定期間のタイムラインを表示できる
- データ種別（目標、日報、フォローアップ、週次フォーカス）でフィルタリングできる
- 目標でフィルタリングでき、特定の目標に関連する活動のみを表示できる
- タイムラインをグルーピング表示できる（日単位、週単位、月単位など）
- 各アクティビティの概要（日付、種別、タイトル、関連目標など）が表示される
- アクティビティをクリックすると、詳細画面に遷移できる

### 次のアクション

- [ ] Phase 1: 設計開始（BL-03, BL-04完了後）
- [ ] PRD作成

### 備考

- **前提条件**: BL-03（日報と目標の関連付け機能）、BL-04（フォローアップと目標の接続機能）、BL-01（目標階層管理機能）が実装されていることが必須
- タイムラインの表示形式は、ユーザーの使いやすさを考慮する（例: カレンダー形式、リスト形式、カード形式など）
- データ量が多い場合のパフォーマンスを考慮する（ページネーション、遅延ロード、仮想スクロールなど）
- グルーピング機能により、データ量が多い場合でも全体像を把握しやすくする
- フィルタリング機能により、必要な情報に絞り込める
- タイムラインの表示項目は、ユーザーがカスタマイズできることが望ましい（任意）
