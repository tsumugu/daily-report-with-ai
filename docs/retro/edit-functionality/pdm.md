# [Role: PdM] 編集機能 振り返り

## Keep（良かった点）

1. **受入基準の明確化**
   - PRDにAC-1からAC-7までの受入基準を明記し、実装の方向性が明確になった
   - E2Eテストで各受入基準をカバーできた

2. **エッジケースの洗い出し**
   - 日報編集時のよかったこと/改善点の削除時の確認ダイアログを要件に含めた
   - エラーハンドリングの仕様を明文化した

3. **メッセージ仕様の明文化**
   - エラーメッセージ、成功メッセージの文言をPRDに明記した
   - Eng/Desが実装時に迷わないようにした

4. **ビジネスルールの明文化**
   - ステータス自動再計算のルールをPRDに明記した
   - 編集権限のルールを明確にした

## Problem（改善点）

1. **Storybookへの反映**
   - 新規コンポーネントのStorybookストーリー作成が実装と同時に行われなかった
   - 受け入れ条件としてStorybookへの反映を必須とすべきだった

2. **E2Eテストの実行タイミング**
   - 実装時に全E2Eテストを実行する必要があるか不明確だった
   - push時にpre-pushフックで全E2Eテストが自動実行されることをドキュメントに明記すべきだった

## Try（次回アクション）

1. **Storybookファーストの徹底**
   - 新規コンポーネント作成時は、FE実装と同時にStorybookストーリーも作成することを必須とする
   - 受け入れ条件にStorybookへの反映を明記する

2. **E2Eテストの実行方針の明確化**
   - 実装時は新規追加したテストのみを実行し、全E2Eテストはpush時に自動実行されることをドキュメントに明記する

3. **受け入れ確認の効率化**
   - 受入基準とE2Eテストの対応関係を明確にし、受け入れ確認を効率化する

## Learnings

- **受入基準の重要性**: 明確な受入基準を設定することで、実装の方向性がブレない
- **E2Eテストの活用**: ユーザーストーリーに基づいてE2Eテストを作成することで、要件充足度を確認できる
- **Storybookへの反映**: 新規コンポーネントのStorybookストーリー作成は実装と同時に行うべき
