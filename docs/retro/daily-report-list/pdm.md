# [Role: PdM] 日報一覧・詳細機能 振り返り

## Keep（良かった点）

1. **PRDの受け入れ要件が明確だった**
   - 「日付降順で一覧表示」「ページング対応」を明確に定義
   - 実装完了時に要件チェックが容易だった

2. **ユーザーストーリーの具体性**
   - 「過去の日報を確認したい」という動機を明確化
   - 開発者が実装の目的を理解しやすかった

3. **段階的リリース計画**
   - Phase 1で一覧・詳細表示、Phase 2で検索・フィルタ
   - スコープを限定して早期にリリース可能

## Problem（改善点）

1. **検索要件の曖昧さ**
   - 「Phase 2で対応」としたが、どの程度の検索が必要か未定義
   - 日付範囲？キーワード検索？優先度を決めておくべきだった

2. **ページングのUX要件不足**
   - 「もっと見る」ボタンか無限スクロールか、PRDで明確にすべきだった
   - 実装者の判断に委ねてしまった

3. **アクセシビリティ要件未定義**
   - キーボード操作、スクリーンリーダー対応の要件がなかった

## Try（次回アクション）

1. **Phase 2要件の事前定義**
   - 検索・フィルタのスコープを次の機能開始前に明確化
   - ユーザーインタビューで優先度を確認

2. **UXパターンの事前決定**
   - ページング方式（ボタン/無限スクロール）をDesと協議して決定
   - PRDに含める

3. **アクセシビリティ要件の追加**
   - 標準的なアクセシビリティ要件をテンプレート化
   - 新機能PRDに含める

## Learnings

### プロダクト観点

- **一覧→詳細のナビゲーション**: ユーザーは一覧から詳細への遷移を自然に行う
- **バッジ表示の価値**: よかったこと/改善点の有無が一目でわかることで、振り返りの動機付けになる

### プロセス観点

- **段階的リリースの効果**: 最小機能でリリースし、フィードバックを得てから拡張する
- **PRDの受け入れ要件**: 明確な要件があると、完了判定がスムーズ

### ユーザー理解

- **日報の活用パターン**: 過去の日報を参照するのは週報作成時や振り返り時
- **一覧性の重要性**: 最近の日報をすぐに確認できることが重要
