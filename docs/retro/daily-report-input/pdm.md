# [Role: PdM] 日報入力機能 振り返り

## Keep（良かった点）

1. **要件の明確化**
   - 入力項目（日付/できごと/学び/よかったこと/改善点）を明確に定義
   - 必須/任意の区別を明確にし、実装の方向性がブレなかった

2. **成功基準の設定**
   - 「ユーザーが毎日日報を書いて保存できる」という明確な基準を設定
   - 受け入れ要件チェックで確認できた

3. **Phase分けの徹底**
   - 下書き保存、タグ入力等をPhase 2に先送り
   - MVPスコープを適切に絞り、機能を完成させられた

4. **UI設計書の詳細化**
   - ワイヤーフレームをASCIIで記載し、イメージを共有できた
   - 状態別表示（ローディング/エラー/成功）を明記

5. **非機能要件の考慮**
   - モバイル/PC両対応を最初から要件に含めた
   - 文字数上限（1000文字）を明確に定義

## Problem（改善点）

1. **トースト通知の仕様未定義**
   - UI設計書に「トースト通知」と記載したが、仕様が曖昧だった
   - 結果的に未実装のまま完了した

2. **離脱確認ダイアログの曖昧さ**
   - 「任意実装」と記載したが、実装すべきか判断基準がなかった
   - MVPで必要かどうかを明確にすべきだった

3. **エラーメッセージの文言未定義**
   - Auth機能の反省を活かせなかった
   - バリデーションエラー時のメッセージ文言を仕様として定義すべきだった

4. **同日重複エラーの仕様未定義**
   - 「同じ日付の日報は登録できない」という仕様がPRDになかった
   - Engが実装時に判断したが、本来はPdMが定義すべきだった

## Try（次回アクション）

1. **トースト通知の仕様明確化**
   - Phase 2 で ToastNotification の仕様を定義
   - 表示位置、表示時間、スタイルを明記

2. **「任意実装」の判断基準策定**
   - MVP / Phase 2 / 将来 の3段階で優先度を明確化
   - 「任意」ではなく「Phase X で実装」と記載

3. **エラーメッセージ仕様の事前定義**
   - PRDにエラーメッセージの文言を含める
   - ユーザーに表示するメッセージは要件として定義する

4. **ビジネスルールの明文化**
   - 「同日重複禁止」等のビジネスルールをPRDに明記
   - 技術的制約ではなく、ビジネス要件として定義する

## Learnings

### 要件定義の学び

- **成功基準の価値**: 明確な基準があることで、完了判定がスムーズになった
- **Phase分けの重要性**: スコープクリープを防ぎ、MVPを完成させられた
- **エラーケースの洗い出し**: 正常系だけでなく、エラーケースもPRDで定義すべき

### プロセスの学び

- **受け入れ要件チェックの有効性**: 実装完了時に要件を1つ1つ確認することで、漏れを防げた
- **振り返りの活用**: Auth機能の反省点（エラーメッセージ未定義）を認識していたが、活かせなかった→次こそ

### チーム連携の学び

- **ロール間の責任分担**: ビジネスルール（同日重複禁止）はPdMが定義すべきだった
- **仕様の曖昧さは後工程に影響**: 「任意」「トースト通知」の曖昧さがそのまま実装に影響した
