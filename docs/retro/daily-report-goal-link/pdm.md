# [Role: PdM] 日報と目標の関連付け機能 振り返り

## Keep（良かった点）

1. **PRDの要件定義が明確だった**
   - ユーザーストーリー（US-1〜US-5）を明確に定義
   - 受け入れ基準を具体的に記載
   - Phase分けを明確に定義（Phase 1で実装する機能を明確化）

2. **段階的リリース計画**
   - Phase 1で基本機能を実装、Phase 2で自動関連付け提案を実装
   - スコープを限定して早期にリリース可能

3. **エッジケースの洗い出し**
   - 削除された目標への対応、目標の期間外の日報への対応などを明確化
   - 実装時の判断が容易だった

## Problem（改善点）

1. **目標選択のUX要件不足**
   - 目標が多数ある場合の選択が煩雑になる可能性がある
   - Phase 2で検索・フィルタ機能を追加する予定だが、Phase 1でも検討すべきだった

2. **関連日報の並び替え要件不足**
   - 目標詳細画面での関連日報表示の並び替え機能が未定義
   - 実装時に追加したが、PRDに含めるべきだった

3. **自動関連付けの要件定義不足**
   - Phase 2で自動関連付け提案を実装する予定だが、具体的な要件が未定義
   - Phase 1完了後に要件定義を進める必要がある

## Try（次回アクション）

1. **UX要件の事前定義**
   - 目標選択の検索・フィルタ機能をPhase 1のPRDに含める
   - 関連日報の並び替え機能をPRDに含める

2. **Phase 2要件の事前定義**
   - 自動関連付け提案の具体的な要件を定義
   - ユーザーインタビューで優先度を確認

3. **エッジケースの追加洗い出し**
   - 実装時に発見したエッジケースをPRDに追加
   - レビュー時にエッジケースを確認するチェックリストを作成

## Learnings

### プロダクト観点

- **関連付けの任意性**: 日報と目標の関連付けを任意にすることで、ユーザーの負担を軽減
- **複数目標への関連付け**: 1つの日報を複数の目標に関連付けられることで、柔軟性を確保
- **目標階層構造との整合性**: 任意の階層レベルの目標に関連付けられることで、柔軟性を確保

### プロセス観点

- **段階的リリースの効果**: 最小機能でリリースし、フィードバックを得てから拡張する
- **PRDの受け入れ要件**: 明確な要件があると、完了判定がスムーズ
- **エッジケースの重要性**: エッジケースを事前に洗い出すことで、実装時の判断が容易

### ユーザー理解

- **日報と目標の関連性**: 日々の活動が目標達成にどう貢献しているかを明確にできる
- **振り返りの効率化**: 目標ごとに日報を集約して表示することで、振り返りを効率化
- **関連付けの負担**: 関連付けの負担を軽減する仕組み（Phase 2で自動関連付け提案）が重要
